# Doc 模块升级说明

更新日期：2026-04-16

## 这次升级解决了什么

这一轮不是单纯把 Docs 做成一个“能写字的页面”，而是把它推进成 Synaply 的结构化协作资产：

- 文档可以携带明确协作语义，而不是只有标题和正文
- Project / Issue / Workflow 都有自己的专门文档入口
- Inbox 和 digest 不再只提示“有文档更新了”，而是知道这是评审包、交接包、决策记录还是发布检查清单
- 团队共享文档与团队内个人私有文档之间的打开路径和上下文恢复更稳定

目标是把 Docs 从“信息存放处”升级成“协作推进器”。

## 当前已落地的能力

### 1. 文档语义化

后端为 Doc 新增了两类核心字段：

- `kind`
- `templateKey`

当前支持的 `DocKind`：

- `GENERAL`
- `PROJECT_BRIEF`
- `DECISION_LOG`
- `REVIEW_PACKET`
- `HANDOFF_PACKET`
- `RELEASE_CHECKLIST`

这意味着同样是文档，系统现在可以区分：

- 这是项目 brief
- 这是一次评审前要交给 reviewer 的 review packet
- 这是一次跨角色移交时的 handoff packet
- 这是发布前的 checklist

后续 Inbox、digest、Project 协作面板、Issue 详情和 Workflow 设置页，都基于这层语义来消费。

### 2. 模板化创建

前端新增了结构化文档创建弹窗，创建时可以直接选择模板：

- 空白文档
- 项目 brief
- 决策记录
- 评审包
- 交接包
- 发布检查清单

每个模板都会自动带入：

- 对应的 `kind`
- 对应的 `templateKey`
- 一套可直接开始填写的默认内容骨架
- 默认标题建议

其中项目 brief 会优先带上项目名，避免出现大量“新文档”。

### 3. Project / Issue / Workflow 的专门文档入口

#### Project

项目详情页的协作区已经补上文档卡片，当前展示三类项目级文档：

- 项目 brief
- 决策记录
- 发布检查清单

用户可以直接：

- 查看最新一份对应类型的文档
- 如果还没有，就从卡片上直接创建

#### Issue

普通 Issue 详情页已经接入 issue 级文档卡片，当前支持：

- 评审包
- 交接包
- 决策记录

这样一条 Issue 在进入 review、handoff 或决策收敛阶段时，不再需要跳出当前上下文去手动找文档位置。

#### Workflow

Workflow 设置面板已经接入 workflow 级文档卡片，当前支持：

- 交接包
- 评审包
- 发布检查清单

同时，这里已经修复了一个关键边界：

- issue 级文档不再双写 `workflowId`
- workflow 卡片只展示真正的 workflow 级文档，不会再被 issue scoped packet 污染

### 4. Docs 页面与上下文路由更稳定

这轮补了一层专门的文档导航逻辑，解决了几个容易出错的点：

- 团队共享文档和团队内个人私有文档使用不同的打开路径
- `team-personal` 文档不会再被错误地打开到团队共享 Docs
- 项目文档子页会继承正确的 docs context，而不是一律压成 `team`
- 打开的文档 tab 仍然能按 workspace / context / project 维度恢复

现在的行为是：

- 团队共享文档默认进入 `/docs`
- 团队内个人私有文档进入 `/personal/doc`
- 项目文档在 `/projects/{projectId}/docs` 下打开
- 如果项目文档属于 `team-personal`，会带上对应 `context` 查询参数

### 5. Inbox 信号与 digest 汇总开始消费 Doc Kind

后端 Inbox 已经开始把结构化文档当成协作信号源，而不只是普通附件。

当前会生成的文档类信号包括：

- `doc.review.ready`
- `doc.handoff.ready`
- `doc.decision.updated`
- `doc.release.updated`
- `digest.generated`

这些信号会结合文档类型、关联 issue / workflow / project 和当前用户上下文，决定：

- 放到 `needs-response`、`needs-attention`、`following` 还是 `digest`
- 是否需要动作
- 默认 action label 是什么

同时 Inbox item metadata 里已经带上：

- `docContext`
- `docKind`
- `docTitle`

这样前端点击 Inbox 条目时，能够直接把用户带到正确的文档上下文，而不是落到错误页面。

digest 当前已支持三种聚合维度：

- issue 级 digest
- workflow 级 digest
- project 级 digest

它们会基于最近一批结构化文档变化，生成“本轮文档协作脉冲”。

## 这一轮真实提升的用户感知

如果从使用者视角看，这轮最明显的变化不是“文档页更漂亮了”，而是下面这几件事：

### 1. 文档不再是孤岛

以前 Docs 更像独立模块；现在它已经开始嵌进：

- Project 协作区
- Issue 详情
- Workflow 设置
- Inbox
- digest

这意味着团队不必先想“去哪写文档”，而是直接在当前协作对象上创建对的文档。

### 2. review / handoff / decision / release 都有了更清晰的承载体

这对 Synaply 很关键，因为产品定位不是“任务列表”，而是“跨角色推进”。

文档的价值不再只是记录，而是承接这些关键协作动作：

- 评审前，准备 review packet
- 移交前，准备 handoff packet
- 决策发生时，沉淀 decision log
- 发布前，维护 checklist

### 3. Inbox 开始具备“文档驱动协作”的能力

以前 Inbox 更容易停留在状态提醒层；现在它已经能告诉你：

- 有一份 review packet ready 了
- 有一份 handoff packet 更新了
- 某个项目的 decision log 变了
- 某个 workflow / project 最近有一批协作文档变化，值得看一眼 digest

这更符合 Synaply 想做的 async collaboration rhythm。

## 当前边界与下一步建议

这轮已经把 Doc 模块从“独立文档编辑”推进到“结构化协作文档”，但还可以继续补满三类能力：

### 1. 服务端进一步收口 workflow 级文档过滤

当前前端已经过滤 issue scoped 文档，issue 新建文档也不再附带 `workflowId`。后续仍建议在服务端补一层明确语义过滤，让 workflow-only query 自身就排除 `issueId` 非空的文档。

### 2. 基于 doc kind 驱动更多专门卡片

现在项目、issue、workflow 都已经有基础卡片，但还可以继续增强：

- 不同 kind 展示不同状态提示
- 直接显示最后更新时间、owner、关联 issue / workflow
- 增加 “需要补齐 / 已准备好” 等更明确的 readiness 提示

### 3. digest 的消费面继续扩大

当前 digest 已经能聚合文档变化，但后续还可以继续接入：

- 首页 pulse
- 项目 sync 页面
- 周报 / 日报输出

这样 Docs 才会真正成为团队节奏的一部分，而不是仅在被主动打开时才有存在感。

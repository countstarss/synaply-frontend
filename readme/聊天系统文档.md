Synaply 前端聊天服务重构需求文档


  1. 概述


  本需求文档旨在详细说明 Synaply 前端聊天服务从当前基于 Convex 的实现迁移至基于
  Supabase Realtime 和 Supabase Storage 的新方案。新方案将支持多种聊天类型，并强
  调本地优先的数据处理原则，同时确保服务器端消息的完整性和权威性。

  2. 核心需求

  2.1 聊天类型


   * 群聊 (Group Chat):
       * 支持创建新的群聊。
       * 支持向现有群聊中添加新成员。
       * 新成员的添加（包括从团队中搜索成员）目前仅需实现前端页面效果和交互逻辑，
         后端 API 待后续实现。
   * 私聊 (Private Chat):
       * 支持用户之间发起一对一的私聊。

  2.2 消息发送与接收


   * 前端直接实现: 消息的发送和接收将直接在前端通过 Supabase Realtime 实现。
   * 用户身份识别: 登录后可获取的 Supabase 用户 session 信息（如
     user.id）将用于标识消息发送者，并确保消息的唯一性。
   * 实时性: 利用 Supabase Realtime 的能力，确保消息的即时传递和更新。

  2.3 本地优先 (Local-First) 原则


   * 本地缓存: 所有聊天消息应缓存到本地存储（例如 IndexedDB
     或类似技术），以实现离线访问和更快的加载速度。
   * 数据同步: 当设备在线时，本地数据应与 Supabase 服务器保持同步。

  2.4 服务器消息完整性与权威性


   * 服务器作为唯一真相来源 (Source of Truth): Supabase
     服务器将始终保存所有聊天消息的最完整、最准确的版本。
   * 本地删除/隐藏:
     用户可以在本地删除或隐藏自己的聊天记录，但这仅影响其个人客户端的显示。
   * 服务器数据不可篡改: 用户无法通过前端操作篡改服务器上的消息记录。其他用户仍能
     获取到完整的、未经修改的消息历史。这意味着即使某个用户在本地删除了消息，该消
     息对于其他参与者来说仍然可见。

  3. 技术考量


   * Supabase Realtime: 用于实现消息的实时推送和订阅。
   * Supabase Storage:
     用于存储聊天相关的媒体文件（如图片、视频等，如果未来有此需求）。
   * Supabase Auth: 利用已有的用户认证体系，获取用户 session
     信息进行身份验证和授权。
   * 本地存储方案: 考虑使用 Dexie.js (IndexedDB wrapper)
     或类似库来实现本地消息缓存，以支持本地优先的特性。
   * 数据模型: 需要定义 Supabase 中聊天消息、群组、私聊等相关的数据表结构。

  4. 待后端 API 支持的功能 (前端仅需 UI/Mockup)


   * 创建群聊: 前端提供创建群聊的
     UI，但实际的群聊创建操作（如在数据库中记录群聊信息）将依赖后端 API。
   * 添加群成员: 前端提供添加成员的
     UI（包括搜索团队成员的界面），但实际的成员添加操作将依赖后端 API。
